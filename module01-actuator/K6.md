# Stage 4. k6 부하테스트 실습 교안

## 수업 목표

이번 실습의 목표는 k6로 Popular Shop API에 부하를 만들고, Prometheus, Grafana, Loki를 함께 사용해 병목을 해석하는 것이다.

학생들은 이 실습이 끝나면 다음 질문에 답할 수 있어야 한다.

- 부하테스트 시나리오는 어떤 운영 질문에서 출발해야 하는가?
- k6에서 VU, iteration, scenario, threshold, check는 각각 무엇인가?
- 부하테스트 결과를 Prometheus, Grafana, Loki에서 어떻게 함께 해석하는가?
- `findPopularProducts` 쿼리는 어떤 조건에서 병목이 되며, 어떤 인덱스와 쿼리 전략을 검토해야 하는가?
- JPA 연관관계 최적화와 집계 쿼리 최적화는 무엇이 다른가?

부하테스트 대상의 핵심 코드는 다음이다.

```java
@Transactional(readOnly = true)
public List<PopularProductResponse> getPopularProducts() {
    shopMetrics.recordPopularProductRequest();
    Timer.Sample sample = shopMetrics.startTimer();
    long startedAt = System.nanoTime();
    LocalDateTime sevenDaysAgo = LocalDateTime.now().minusDays(7);
    try {
        List<PopularProductResponse> popularProducts =
                productRepository.findPopularProducts(sevenDaysAgo, PageRequest.of(0, 5));

        log.info(
                "event=popular_products_queried resultCount={} durationMs={}",
                popularProducts.size(),
                elapsedMillis(startedAt)
        );

        return popularProducts;
    } finally {
        shopMetrics.stopPopularProductQueryTimer(sample);
    }
}
```

## 1. 부하테스트 전에 정해야 하는 것

부하테스트는 도구 실행이 아니라 질문을 검증하는 과정이다.

먼저 다음을 정한다.

| 항목 | 예시 |
| --- | --- |
| 테스트 목적 | 인기 상품 조회 API가 주문 트래픽과 함께 증가할 때 버티는지 확인 |
| 대상 API | `GET /products/popular`, `POST /orders` |
| 사용자 행동 | 사용자는 인기 상품을 보고 일부 사용자는 주문한다 |
| 트래픽 비율 | 인기 상품 조회 80%, 주문 생성 20% |
| 목표 부하 | 1분 동안 10 VU, 이후 50 VU까지 증가 |
| 성공 기준 | p95 응답 시간 500ms 이하, 실패율 1% 미만 |
| 관찰 지표 | HTTP latency, error rate, RPS, DB query duration, Loki slow logs |
| 종료 기준 | error rate 급증, DB CPU 급등, p95 지연 악화 |

수업 포인트:

- 막연히 많은 요청을 보내는 것은 부하테스트가 아니다.
- 운영에서 발생 가능한 사용자 행동을 단순화한 모델이 시나리오다.
- 시나리오는 항상 가설, 입력, 관찰 지표, 성공 기준을 포함해야 한다.

## 2. 가상 시나리오 작성 방법

### 2.1 시나리오 구성 요소

좋은 부하테스트 시나리오는 다음 요소를 가진다.

| 구성 요소 | 설명 | 예시 |
| --- | --- | --- |
| 사용자 유형 | 어떤 사용자인가 | 상품을 둘러보는 사용자, 주문하는 사용자 |
| 행동 순서 | 어떤 API를 어떤 순서로 호출하는가 | 인기 상품 조회 후 주문 생성 |
| 행동 비율 | 전체 트래픽 중 각 행동 비중 | 조회 80%, 주문 20% |
| 대기 시간 | 실제 사용자처럼 요청 사이에 쉬는 시간 | 1~3초 sleep |
| 데이터 분포 | 어떤 상품/주문 데이터를 사용하는가 | productId 1~7 랜덤 |
| 실패 조건 | 어떤 경우 실패로 볼 것인가 | status 400/500, p95 초과 |
| 관찰 기준 | 어떤 지표를 볼 것인가 | `http_req_duration`, `shop.products.popular.query.duration` |

### 2.2 나쁜 시나리오와 좋은 시나리오

나쁜 시나리오:

```text
/products/popular를 최대한 많이 호출한다.
```

문제:

- 왜 이 API를 호출하는지 운영 맥락이 없다.
- 어느 정도의 부하가 정상인지 기준이 없다.
- 주문 트래픽과 함께 발생하는 병목을 볼 수 없다.
- 성공/실패 기준이 없다.

좋은 시나리오:

```text
프로모션 배너 노출 후 사용자의 80%는 인기 상품을 조회하고, 20%는 주문을 시도한다.
5분 동안 VU를 5명에서 50명까지 증가시킨다.
성공 기준은 전체 요청 실패율 1% 미만, /products/popular p95 500ms 이하이다.
부하 중 Prometheus에서 HTTP latency와 custom timer를 확인하고, Loki에서 popular_products_queried 로그의 durationMs를 확인한다.
```

## 3. 가상 시나리오 프롬프트 작성법

AI를 사용해 부하테스트 시나리오 초안을 만들 때는 다음 정보를 반드시 포함한다.

```text
너는 성능 테스트 엔지니어다.
Spring Boot 쇼핑몰 API에 대해 k6 부하테스트 시나리오를 설계해줘.

서비스 상황:
- 사용자는 인기 상품 목록을 자주 조회한다.
- 일부 사용자는 인기 상품을 보고 주문을 생성한다.
- 인기 상품 조회는 최근 7일 주문 데이터를 group by 해서 Top 5 상품을 반환한다.

API:
- GET /products/popular
- POST /orders body: {"items":[{"productId":1,"quantity":1}]}
- GET /products/{id}

목표:
- 조회 중심 트래픽 80%, 주문 트래픽 20%
- 5분 동안 점진적으로 VU 증가
- p95 응답시간 500ms 이하
- 실패율 1% 미만

요구사항:
- k6 script 예시를 작성해줘.
- thresholds, checks, sleep을 포함해줘.
- Prometheus/Grafana/Loki에서 확인할 지표와 로그 쿼리도 제안해줘.
```

프롬프트 작성 원칙:

| 원칙 | 설명 |
| --- | --- |
| 시스템 역할을 준다 | 성능 테스트 엔지니어, SRE, 백엔드 엔지니어 등 |
| 서비스 맥락을 준다 | 어떤 이벤트 또는 운영 상황인지 설명 |
| API 계약을 준다 | Method, URI, request body, expected status |
| 부하 모델을 준다 | VU, duration, ramp-up, traffic ratio |
| 성공 기준을 준다 | p95, error rate, RPS, threshold |
| 관찰 도구를 준다 | k6, Prometheus, Grafana, Loki |
| 제약을 준다 | 로컬 환경, MySQL, Spring Boot 등 |

## 4. k6 핵심 개념

| 개념 | 설명 |
| --- | --- |
| VU | Virtual User. 동시에 행동하는 가상 사용자 |
| iteration | VU가 `default function` 또는 scenario function을 한 번 실행한 횟수 |
| scenario | 어떤 실행 모델로 어떤 함수를 실행할지 정의한 단위 |
| executor | 부하 패턴. `constant-vus`, `ramping-vus`, `constant-arrival-rate` 등 |
| check | 응답이 기대 조건을 만족하는지 검증 |
| threshold | 테스트 전체 성공/실패를 판단하는 기준 |
| sleep | 실제 사용자 행동처럼 요청 사이에 대기 |
| Trend | 시간값 분포를 수집하는 커스텀 메트릭 |
| Rate | 성공/실패 비율을 수집하는 커스텀 메트릭 |
| Counter | 누적 횟수를 수집하는 커스텀 메트릭 |

## 5. k6 주요 지표

k6 결과에서 우선 볼 지표는 다음이다.

| 지표 | 의미 | 해석 |
| --- | --- | --- |
| `http_reqs` | 전체 HTTP 요청 수 | 총 트래픽 규모 |
| `http_req_duration` | 요청 전체 시간 | 가장 중요한 사용자 체감 지연 시간 |
| `http_req_failed` | 실패한 요청 비율 | 4xx/5xx 또는 check 실패 여부 확인 |
| `http_req_waiting` | 서버 응답 대기 시간, TTFB | 서버 처리 지연에 가까운 값 |
| `http_req_connecting` | TCP 연결 시간 | 네트워크/커넥션 문제 확인 |
| `http_req_blocked` | 커넥션 슬롯 대기 시간 | 연결 재사용/커넥션 병목 확인 |
| `iterations` | 시나리오 반복 횟수 | VU가 실제로 수행한 작업량 |
| `vus` | 현재 활성 VU | 부하 증가 상태 확인 |
| `checks` | check 성공률 | 응답 검증 성공률 |
| `data_received` | 받은 데이터 양 | 응답 크기 변화 확인 |
| `data_sent` | 보낸 데이터 양 | 요청 body 규모 확인 |

수업에서 강조할 해석:

- `http_req_duration` p95가 증가하면 사용자가 느끼는 응답 시간이 나빠진 것이다.
- `http_req_waiting`이 증가하면 서버 내부 처리 또는 DB 처리 지연 가능성이 크다.
- `http_req_failed`가 증가하면 latency보다 먼저 안정성 문제를 봐야 한다.
- 평균값보다 p90, p95, p99가 중요하다.
- k6 지표만으로 원인을 확정하지 말고 Prometheus와 Loki로 좁힌다.

해석 기준:

| 지표 | 의미 | 수업에서의 사용 |
| --- | --- | --- |
| p90 | 90% 요청이 이 시간 이하 | 전반적인 안정성 확인 |
| p95 | 95% 요청이 이 시간 이하 | 주요 threshold 기준 |
| p99 | 99% 요청이 이 시간 이하 | 극단적으로 느린 tail latency 확인 |

p99를 threshold로 바로 쓰지 않는 이유:

- 짧은 로컬 실습에서는 요청 표본 수가 적어 p99가 크게 흔들릴 수 있다.
- outlier 한두 개가 전체 테스트를 실패로 만들 수 있다.
- 수업에서는 p95를 합격 기준으로 두고 p99는 병목 의심 신호로 해석한다.
- 운영 수준의 긴 테스트에서는 p99 threshold를 별도로 둘 수 있다.

## 5.1 p90, p95, p99 이해하기

`p90`, `p95`, `p99`는 평균이 아니라 백분위수(percentile) 지표다.
성능 테스트에서는 평균 응답 시간보다 백분위수를 우선해서 봐야 한다.
평균은 일부 매우 느린 요청에 흔들리거나, 반대로 일부 사용자가 겪는 심각한 지연을 숨길 수 있기 때문이다.

예를 들어 `p95 = 500ms`라면 다음 뜻이다.

```text
전체 요청 중 95%는 500ms 이하로 응답했다.
나머지 5%는 500ms보다 느렸다.
```

각 지표의 의미는 다음과 같다.

| 지표 | 의미 | 실무 해석 |
| --- | --- | --- |
| `p50` | 50% 요청이 이 시간 이하 | 중앙값. 일반적인 요청의 체감 속도 |
| `p90` | 90% 요청이 이 시간 이하 | 대부분의 사용자가 경험하는 성능 |
| `p95` | 95% 요청이 이 시간 이하 | 성능 테스트와 SLO에서 자주 쓰는 합격 기준 |
| `p99` | 99% 요청이 이 시간 이하 | tail latency. 극단적으로 느린 요청 확인 |

예시:

```text
요청 100개 중
90개는 300ms 이하
5개는 800ms 이하
4개는 2s 이하
1개는 5s
```

이 경우 대략 다음처럼 해석할 수 있다.

```text
p90 = 300ms
p95 = 800ms
p99 = 5s 근처
```

평균만 보면 서비스가 괜찮아 보일 수 있다.
하지만 `p99`가 5초라면 일부 사용자는 명확하게 느린 응답을 경험한다.
트래픽이 큰 서비스에서는 1%도 작은 숫자가 아니다.
하루 요청이 1,000만 건이면 p99 밖의 1%는 10만 건이다.

기본 해석 기준:

| 상황 | 해석 |
| --- | --- |
| 평균은 낮고 p95가 높다 | 일부 요청이 느려져 사용자 경험이 갈라지고 있다 |
| p95와 p99가 함께 증가한다 | 전반적인 처리 지연 또는 병목 가능성이 크다 |
| p99만 튄다 | GC, DB lock, 네트워크 지연, 커넥션 대기 같은 outlier 가능성을 본다 |
| p95는 좋지만 실패율이 높다 | 빠르게 실패하는 것이므로 성능 성공으로 볼 수 없다 |

주의할 점:

- 짧은 로컬 테스트에서는 요청 수가 적어 `p99`가 크게 흔들릴 수 있다.
- 표본이 적을수록 `p99`는 가장 느린 요청 한두 개에 가까워진다.
- 여러 서버의 p95를 단순 평균 내면 전체 p95가 되지 않는다.
- percentile은 원본 분포 또는 histogram 기반으로 집계해야 한다.

## 5.2 성공 기준 수립 방법

부하테스트 성공 기준은 단순히 "빠르면 성공"이 아니다.
성공 기준은 대상 API, 트래픽 조건, 측정 기간, 응답 시간, 실패율, 리소스 한계를 함께 정의해야 한다.

나쁜 기준:

```text
평균 응답 시간이 500ms 이하면 성공
```

좋은 기준:

```text
동시 사용자 50명, 5분 동안 조회 80% / 주문 20% 트래픽에서
/products/popular p95 응답 시간이 500ms 이하이고,
전체 실패율이 1% 미만이면 성공
```

성공 기준을 세우는 순서는 다음과 같다.

| 단계 | 질문 | 예시 |
| --- | --- | --- |
| 1 | 어떤 사용자 행동을 검증하는가 | 인기 상품 조회 후 일부 사용자가 주문한다 |
| 2 | 어떤 API가 핵심 경로인가 | `GET /products/popular`, `POST /orders` |
| 3 | 어느 정도 부하를 줄 것인가 | 5분 동안 10 VU에서 50 VU까지 증가 |
| 4 | 어떤 지표로 성공을 판단하는가 | p95, p99, 실패율, RPS |
| 5 | 어떤 리소스 한계를 넘으면 위험한가 | CPU, heap, Hikari connection, DB query duration |
| 6 | 어떤 조건이면 테스트를 중단하는가 | 5xx 급증, 실패율 급증, DB 연결 고갈 |

API 유형별 기준은 다르게 잡아야 한다.
모든 API에 같은 p95 기준을 적용하면 실제 운영 중요도를 반영하기 어렵다.

| API 유형 | 기준 예시 |
| --- | --- |
| 단순 조회 API | `p95 < 300ms`, `p99 < 1s` |
| 집계/검색 API | `p95 < 500~800ms`, `p99 < 2s` |
| 주문/결제 API | `p95 < 800ms~1s`, `p99 < 3s` |
| 외부 연동 API | `p95 < 2s`, `p99 < 5s` |
| 비동기 작업 | 응답 시간보다 큐 적체와 처리 지연 시간 기준 사용 |

이 실습에서는 로컬 환경과 작은 데이터셋을 고려해 다음 기준을 기본값으로 사용한다.

```text
공통 성공 기준:
- 전체 http_req_failed < 1%
- 전체 http_req_duration p95 < 1000ms

/products/popular 기준:
- http_req_duration{type:popular} p95 < 500ms
- p99는 합격/불합격보다 병목 의심 신호로 관찰

/orders 기준:
- http_req_duration{type:order} p95 < 1000ms
- 재고 부족 같은 기대 가능한 비즈니스 실패와 5xx 장애를 구분
```

운영 수준 기준으로 확장하면 다음 요소를 추가한다.

```text
운영형 성공 기준 예시:
- 피크 트래픽 30분 동안 /products/popular p95 < 500ms
- 피크 트래픽 30분 동안 /products/popular p99 < 2s
- 전체 HTTP 실패율 < 0.1%
- HTTP 5xx 비율 < 0.05%
- 애플리케이션 CPU 평균 < 70%
- JVM heap 사용률 < 80%
- Hikari active connection 사용률 < 80%
- 인기 상품 쿼리 평균 시간이 HTTP p95 증가와 함께 악화되지 않을 것
```

k6 threshold로 표현하면 다음과 같다.

```javascript
export const options = {
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<1000'],
    'http_req_duration{type:popular}': ['p(95)<500'],
    'http_req_duration{type:order}': ['p(95)<1000'],
  },
};
```

수업에서 강조할 판단 원칙:

- 평균이 아니라 p95를 기본 합격 기준으로 둔다.
- p99는 tail latency를 보기 위한 관찰 지표로 먼저 사용한다.
- 실패율이 높으면 latency가 낮아도 성공이 아니다.
- 성능 기준은 반드시 트래픽 조건과 측정 기간을 포함해야 한다.
- 비즈니스 실패와 시스템 실패를 분리해서 해석해야 한다.
- k6 결과만 보지 말고 Prometheus 메트릭과 Loki 로그로 원인을 좁혀야 한다.

## 6. thresholds와 checks

### 6.1 check

`check`는 개별 응답이 기대 조건을 만족하는지 확인한다.

```javascript
check(res, {
  'popular status is 200': (r) => r.status === 200,
  'popular response is not empty': (r) => r.json().length > 0,
});
```

check 실패는 원인 분석 신호다. 테스트 자체를 실패시키려면 threshold와 함께 사용한다.

### 6.2 threshold

`threshold`는 테스트 전체의 합격 기준이다.

```javascript
export const options = {
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<500'],
    'http_req_duration{type:popular}': ['p(95)<500'],
    'http_req_duration{type:order}': ['p(95)<1000'],
  },
};
```

해석:

| threshold | 의미 |
| --- | --- |
| `http_req_failed: rate<0.01` | 실패율 1% 미만 |
| `http_req_duration: p(95)<500` | 전체 요청 p95 500ms 미만 |
| `http_req_duration{type:popular}` | `type=popular` 태그가 붙은 요청만 기준 적용 |
| `http_req_duration{type:order}` | 주문 요청만 별도 기준 적용 |

## 8. 실습 2: 조회 80%, 주문 20% 혼합 부하

목적:

실제 사용자 흐름에 더 가까운 혼합 트래픽을 만든다.

파일명:

```text
k6/scripts/02-mixed-shop-traffic.js
```

스크립트:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8080';

export const options = {
  scenarios: {
    mixed_shop_traffic: {
      executor: 'ramping-vus',
      stages: [
        { duration: '30s', target: 10 },
        { duration: '1m', target: 30 },
        { duration: '1m', target: 50 },
        { duration: '30s', target: 0 },
      ],
    },
  },
  thresholds: {
    http_req_failed: ['rate<0.05'],
    'http_req_duration{type:popular}': ['p(95)<700'],
    'http_req_duration{type:order}': ['p(95)<1200'],
  },
};

export default function () {
  const traffic = Math.random();

  if (traffic < 0.8) {
    getPopularProducts();
  } else {
    createOrder();
  }

  sleep(Math.random() * 2 + 1);
}

function getPopularProducts() {
  const res = http.get(`${BASE_URL}/products/popular`, {
    tags: { type: 'popular' },
  });

  check(res, {
    'popular status is 200': (r) => r.status === 200,
  });
}

function createOrder() {
  const productId = Math.floor(Math.random() * 7) + 1;
  const body = JSON.stringify({
    items: [{ productId, quantity: 1 }],
  });

  const res = http.post(`${BASE_URL}/orders`, body, {
    headers: { 'Content-Type': 'application/json' },
    tags: { type: 'order' },
  });

  check(res, {
    'order status is 201 or stock failure': (r) => r.status === 201 || r.status === 400,
  });
}
```

수업 포인트:

- 현재 `data.sql`의 상품 재고는 100개씩이다.
- 주문 부하를 길게 주면 재고 부족으로 400 응답이 발생할 수 있다.
- 이 400은 시스템 장애가 아니라 비즈니스 실패일 수 있다.
- threshold에서 모든 400을 실패로 볼지, 재고 부족을 정상 비즈니스 응답으로 볼지 정해야 한다.

## 9. 실습 3: arrival-rate 기반 부하

VU 수를 늘리는 방식은 사용자를 늘리는 모델이다.
요청량 자체를 일정하게 만들고 싶으면 `constant-arrival-rate`를 사용한다.

파일명:

```text
k6/scripts/03-popular-arrival-rate.js
```

스크립트:

```javascript
import http from 'k6/http';
import { check } from 'k6';

const BASE_URL = __ENV.BASE_URL || 'http://localhost:8080';

export const options = {
  scenarios: {
    popular_fixed_rate: {
      executor: 'constant-arrival-rate',
      rate: 30,
      timeUnit: '1s',
      duration: '2m',
      preAllocatedVUs: 20,
      maxVUs: 100,
    },
  },
  thresholds: {
    http_req_failed: ['rate<0.01'],
    'http_req_duration{type:popular}': ['p(95)<500'],
  },
};

export default function () {
  const res = http.get(`${BASE_URL}/products/popular`, {
    tags: { type: 'popular' },
  });

  check(res, {
    'status is 200': (r) => r.status === 200,
  });
}
```

해석:

- `rate: 30`, `timeUnit: '1s'`는 초당 30회 시작을 목표로 한다.
- 서버가 느려지면 더 많은 VU가 필요하다.
- `maxVUs`에 도달하면 k6가 목표 요청률을 유지하지 못할 수 있다.
- 운영 목표가 RPS라면 arrival-rate 모델이 더 적합하다.

```logql
  | regexp "durationMs=([5-9][0-9]{2}|[1-9][0-9]{3,})"
```

### API 오류 로그

```logql
{job="popular-shop"} |= "event=api_error"
```

### 주문 실패 로그

```logql
{job="popular-shop"} |= "event=order_create_failed"
```

### traceId로 특정 요청 추적

느린 요청 로그에서 `traceId`를 복사한다.

```text
traceId=복사한-trace-id
```

같은 요청의 전체 로그 조회:

```logql
{job="popular-shop"} |= "traceId=복사한-trace-id"
```

수업 포인트:

- Prometheus는 수치 변화와 추세를 빠르게 보여준다.
- Loki는 특정 요청의 로그와 원인을 좁히는 데 사용한다.
- `traceId`는 요청마다 달라지는 값이므로 Loki label로 만들지 않는다.
- `durationMs`도 label이 아니라 로그 본문에서 추출한다.

## 13. Grafana 대시보드 권장 구성

부하테스트 전용 대시보드는 다음 패널로 시작한다.

| 패널 제목 | Data source | Query |
| --- | --- | --- |
| HTTP RPS | Prometheus | `sum by (uri) (rate(http_server_requests_seconds_count[1m]))` |
| HTTP Error Rate | Prometheus | `sum(rate(http_server_requests_seconds_count{status=~"4..|5.."}[1m])) / sum(rate(http_server_requests_seconds_count[1m]))` |
| HTTP Avg Duration | Prometheus | `sum by (uri) (rate(http_server_requests_seconds_sum[1m])) / sum by (uri) (rate(http_server_requests_seconds_count[1m]))` |
| Popular Product RPS | Prometheus | `rate(shop_products_popular_requests_total[1m])` |
| Popular Product Query Avg | Prometheus | `rate(shop_products_popular_query_duration_seconds_sum[1m]) / rate(shop_products_popular_query_duration_seconds_count[1m])` |
| Order Failure by Reason | Prometheus | `sum by (reason) (rate(shop_orders_failed_total[1m]))` |
| Recent Popular Product Logs | Loki | `{job="popular-shop"} |= "event=popular_products_queried"` |
| Slow Request Logs | Loki | `{job="popular-shop"} |= "event=request_completed" | regexp "durationMs=([5-9][0-9]{2}|[1-9][0-9]{3,})"` |
| API Error Logs | Loki | `{job="popular-shop"} |= "event=api_error"` |

### 13.1 Grafana 변수 권장 구성

대시보드 상단에 변수를 만들면 API별 필터링이 쉬워진다.

| 변수명 | Data source | Query | 용도 |
| --- | --- | --- | --- |
| `uri` | Prometheus | `label_values(http_server_requests_seconds_count, uri)` | URI별 HTTP 지표 필터 |
| `method` | Prometheus | `label_values(http_server_requests_seconds_count, method)` | HTTP method 필터 |
| `status` | Prometheus | `label_values(http_server_requests_seconds_count, status)` | HTTP status 필터 |
| `level` | Loki | `label_values({job="popular-shop"}, level)` | 로그 level 필터 |

수업에서는 변수를 꼭 만들 필요는 없지만, 대시보드가 커질수록 `uri`, `method`, `status` 변수가 유용하다.

### 13.4 대시보드 행 구성 예시

Grafana Dashboard는 다음 행(row)으로 구성하면 수업 흐름을 설명하기 쉽다.

| Row | 목적 | 패널 |
| --- | --- | --- |
| Traffic | 부하가 의도대로 들어왔는지 확인 | Total HTTP RPS, HTTP RPS by URI, Popular Product RPS |
| Latency | 어떤 구간이 느려졌는지 확인 | HTTP Avg Duration by URI, HTTP Max Duration by URI, Popular Product Query Avg Duration |
| Error | 실패가 증가했는지 확인 | HTTP Error Rate, HTTP Status Count by URI, API Error by Reason, Order Failure by Reason |
| Resource | 서버/DB 주변 병목 힌트 확인 | JVM Heap Memory Used, Process CPU Usage, Hikari Active Connections |
| Logs | 수치 변화의 원인 추적 | Recent Popular Product Logs, Slow HTTP Request Logs, API Error Logs, Order Failure Logs |

중요한 제한:

- 현재 구성은 k6 자체 메트릭을 Prometheus로 보내지 않는다.
- k6의 `p95`, `p99`, `http_req_failed`는 k6 콘솔과 `k6/results/*-summary.md`에서 확인한다.
- Grafana에서는 Spring Boot가 노출한 서버 측 Prometheus 메트릭과 Loki 로그를 중심으로 본다.
- 운영 수준에서 k6 메트릭까지 Grafana에 함께 그리고 싶다면 k6 Prometheus remote write 구성을 별도로 추가해야 한다.

대시보드 해석 순서:

1. k6에서 p95와 실패율을 확인한다.
2. Grafana에서 HTTP RPS가 의도한 만큼 들어왔는지 확인한다.
3. HTTP duration이 증가한 URI를 찾는다.
4. custom metric에서 인기 상품 쿼리 시간이 함께 증가했는지 확인한다.
5. Loki에서 `event=popular_products_queried`의 `durationMs`와 traceId를 확인한다.
6. 오류가 있다면 `event=api_error`, `event=order_create_failed`를 확인한다.
7. DB 쿼리와 인덱스 전략으로 병목 가설을 정리한다.

## 14. 부하테스트 결과 해석 예시

### 상황 1. `/products/popular` p95만 증가

관찰:

```text
k6: http_req_duration p95 증가
Prometheus: /products/popular 평균 응답 시간 증가
Prometheus: shop.products.popular.query.duration 증가
Loki: event=popular_products_queried durationMs 증가
```

해석:

- 인기 상품 집계 쿼리가 병목일 가능성이 크다.
- DB 인덱스, group by, order by, 집계 대상 row 수를 확인한다.

### 상황 2. 주문 API도 함께 느려짐

관찰:

```text
/products/popular도 느려지고 /orders도 느려짐
shop.payment.duration은 일정하지만 shop.orders.creation.duration 증가
order_create_failed 로그 증가
```

해석:

- DB connection, transaction, row lock, 재고 감소 경합 가능성을 본다.
- 단순히 인기 상품 쿼리만의 문제가 아닐 수 있다.

### 상황 3. k6 실패율 증가, Loki에는 validation 오류 증가

관찰:

```text
k6: http_req_failed 증가
Prometheus: shop.api.errors{reason="validation"} 증가
Loki: event=api_error reason=validation 증가
```

해석:

- 시스템 장애가 아니라 테스트 데이터 또는 request body가 잘못되었을 수 있다.
- 부하테스트 스크립트의 데이터 생성 로직을 먼저 확인한다.

### 상황 4. 주문 실패가 대부분 insufficient_stock

관찰:

```text
Prometheus: shop.orders.failed{reason="insufficient_stock"} 증가
Loki: event=order_create_failed exceptionType=InsufficientStockException 증가
```

해석:

- 재고가 제한된 테스트 데이터에서 자연스럽게 발생한 비즈니스 실패다.
- 성능 실패로 볼지, 기대 가능한 비즈니스 실패로 볼지 기준을 분리해야 한다.

## 15. `findPopularProducts` 병목 분석

현재 쿼리의 형태:

```sql
select
    p.id,
    p.name,
    p.price,
    count(oi.id)
from order_items oi
join products p on oi.product_id = p.id
join orders o on oi.order_id = o.id
where o.created_at >= ?
group by p.id, p.name, p.price
order by count(oi.id) desc, p.id asc
limit 5;
```

병목 가능 지점:

| 지점 | 이유 |
| --- | --- |
| `orders.created_at` 필터 | 최근 7일 주문만 골라야 하므로 범위 검색이 필요 |
| `order_items.order_id` 조인 | 필터링된 주문의 주문 항목을 찾아야 함 |
| `order_items.product_id` 조인 | 상품별 group by 기준 |
| `group by p.id, p.name, p.price` | 집계 대상 row가 많으면 비용 증가 |
| `order by count(oi.id) desc` | 집계 결과 정렬 필요 |
| `limit 5` | 정렬 이후 상위 5개만 반환하므로 집계 비용 자체를 없애지는 못함 |

중요한 해석:

- `limit 5`가 있어도 DB는 최근 7일 주문 항목을 집계해야 한다.
- `order by count(...)`는 일반 인덱스로 바로 해결하기 어렵다.
- 인덱스는 필터와 조인 범위를 줄이는 데 우선 사용된다.
- 데이터가 커지면 실시간 집계 대신 사전 집계 또는 캐시가 필요할 수 있다.

## 16. indexing 전략

현재 schema에는 foreign key는 있지만 조회 최적화를 위한 명시적 보조 인덱스가 없다.

```sql
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    created_at DATETIME(6) NOT NULL
);

CREATE TABLE order_items (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL
);
```

### 16.1 기본 후보 인덱스

최근 7일 주문 필터:

```sql
CREATE INDEX idx_orders_created_at_id
ON orders (created_at, id);
```

주문에서 주문 항목으로 조인:

```sql
CREATE INDEX idx_order_items_order_id_product_id
ON order_items (order_id, product_id);
```

상품 기준 집계 또는 역방향 접근:

```sql
CREATE INDEX idx_order_items_product_id_order_id
ON order_items (product_id, order_id);
```

수업 포인트:

- `orders(created_at, id)`는 최근 주문 범위를 빠르게 찾고, 이어서 order id를 조인에 사용하기 위한 후보이다.
- `order_items(order_id, product_id)`는 필터링된 주문 id로 주문 항목을 찾는 경로에 적합하다.
- `order_items(product_id, order_id)`는 상품 기준으로 집계하거나 다른 쿼리에서 product id를 먼저 사용할 때 후보이다.
- 모든 인덱스를 무작정 추가하면 쓰기 비용과 저장 공간이 증가한다.
- 실제 선택은 `EXPLAIN`과 부하테스트 결과로 판단한다.

### 16.2 EXPLAIN으로 확인

MySQL에서 실행:

```sql
EXPLAIN
select
    p.id,
    p.name,
    p.price,
    count(oi.id)
from order_items oi
join products p on oi.product_id = p.id
join orders o on oi.order_id = o.id
where o.created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
group by p.id, p.name, p.price
order by count(oi.id) desc, p.id asc
limit 5;
```

확인할 것:

| 항목 | 의미 |
| --- | --- |
| `type` | 접근 방식. `ALL`이면 full scan 가능성 |
| `key` | 실제 사용된 인덱스 |
| `rows` | 예상 스캔 row 수 |
| `Extra` | `Using temporary`, `Using filesort` 여부 |

해석:

- `Using temporary`, `Using filesort`는 집계/정렬 쿼리에서 자주 나타난다.
- 이 자체가 항상 문제는 아니다.
- 문제는 부하 상황에서 row 수와 실행 시간이 감당 가능한지다.

### 16.3 인덱스 적용 후 비교 절차

1. 인덱스 적용 전 k6 baseline을 실행한다.
2. Prometheus에서 `shop.products.popular.query.duration` 평균을 기록한다.
3. Loki에서 `popular_products_queried durationMs`의 평균과 느린 로그를 기록한다.
4. MySQL `EXPLAIN` 결과를 저장한다.
5. 후보 인덱스를 적용한다.
6. 같은 k6 스크립트를 같은 조건으로 다시 실행한다.
7. k6 p95, Prometheus query duration, Loki durationMs를 비교한다.
8. 쓰기 API인 `/orders`의 지연이 악화되지 않았는지 확인한다.

## 17. Query 최적화 전략

### 17.1 현재 JPQL DTO projection 유지

현재 쿼리는 Entity를 가져오지 않고 DTO로 바로 조회한다.

장점:

- 화면에 필요한 컬럼만 가져온다.
- Entity graph나 fetch join보다 집계 조회에 적합하다.
- N+1 문제가 발생하기 어렵다.

한계:

- 실시간 집계 비용은 그대로 존재한다.
- 데이터가 많아지면 group by와 order by가 부담된다.

### 17.2 count 기준 재검토

현재 쿼리는 `count(oi.id)`를 사용한다.
즉, 판매 수량이 아니라 주문 항목 줄 수를 기준으로 인기 상품을 계산한다.

상품 1개를 10개 주문해도 주문 항목 1줄이면 count는 1이다.
판매 수량 기준이 목적이면 다음처럼 `sum(oi.quantity)`가 더 적합하다.

```java
@Query("""
        select new com.wanted.actuator.product.dto.PopularProductResponse(
            p.id, p.name, p.price, sum(oi.quantity)
        )
        from OrderItem oi
        join oi.product p
        join oi.order o
        where o.createdAt >= :startDate
        group by p.id, p.name, p.price
        order by sum(oi.quantity) desc, p.id asc
        """)
List<PopularProductResponse> findPopularProducts(
        @Param("startDate") LocalDateTime startDate,
        Pageable pageable
);
```

수업 질문:

- 인기 상품의 기준은 주문 건수인가, 판매 수량인가?
- 비즈니스 정의가 바뀌면 성능 쿼리도 바뀌는가?

### 17.3 집계 테이블 전략

트래픽이 커지면 매 요청마다 최근 7일 데이터를 실시간 group by 하지 않는 전략을 검토한다.

예시 테이블:

```sql
CREATE TABLE product_sales_daily (
    sales_date DATE NOT NULL,
    product_id BIGINT NOT NULL,
    order_count BIGINT NOT NULL,
    quantity_sum BIGINT NOT NULL,
    PRIMARY KEY (sales_date, product_id),
    INDEX idx_product_sales_daily_product_date (product_id, sales_date)
);
```

조회 예시:

```sql
select
    p.id,
    p.name,
    p.price,
    sum(ps.quantity_sum) as sales_count
from product_sales_daily ps
join products p on ps.product_id = p.id
where ps.sales_date >= current_date - interval 7 day
group by p.id, p.name, p.price
order by sales_count desc, p.id asc
limit 5;
```

장점:

- 원본 `orders`, `order_items` 전체를 매번 집계하지 않는다.
- 최근 7일이면 최대 `7일 x 상품 수` row만 집계한다.
- 인기 상품 API 응답 시간이 안정적이다.

단점:

- 데이터 정합성, 배치/이벤트 처리, 재집계 로직이 필요하다.
- 실시간성이 조금 떨어질 수 있다.

### 17.4 캐시 전략

인기 상품은 모든 사용자에게 거의 같은 결과를 반환한다.
따라서 짧은 TTL 캐시가 효과적일 수 있다.

예시 전략:

| 전략 | 설명 |
| --- | --- |
| Local Cache | 단일 인스턴스 실습 환경에서 간단히 적용 가능 |
| Redis Cache | 다중 인스턴스 환경에서 공유 캐시로 사용 |
| TTL 10~60초 | 인기 상품은 초 단위 실시간성이 필요 없는 경우가 많음 |
| Cache Aside | miss 시 DB 조회 후 캐시 저장 |

주의:

- 캐시는 느린 쿼리를 숨길 수 있지만 쿼리 자체를 개선하지는 않는다.
- 캐시 hit ratio를 메트릭으로 함께 봐야 한다.
- 부하테스트는 cache warm-up 전/후를 분리해서 실행해야 한다.

## 18. 연관관계 최적화 전략

`findPopularProducts`는 DTO projection 집계 쿼리이므로 일반적인 N+1 문제와는 다르다.
그래도 수업에서는 집계 쿼리 최적화와 연관관계 조회 최적화를 구분해야 한다.

현재 연관관계:

```java
// OrderItem
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "order_id")
private Order order;

@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "product_id")
private Product product;

// Order
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
private List<OrderItem> items = new ArrayList<>();

// Product
@OneToMany(mappedBy = "product", fetch = FetchType.LAZY)
private List<OrderItem> orderItems = new ArrayList<>();
```

좋은 점:

- `ManyToOne`과 `OneToMany`가 `LAZY`다.
- 기본 조회에서 불필요한 연관 데이터를 즉시 로딩하지 않는다.

주의할 점:

- `OrderResponse.from(order)`가 내부에서 `order.getItems()`를 순회하면 주문 단건 조회에서 item 조회가 추가로 발생할 수 있다.
- 여러 주문을 목록으로 조회하는 기능이 생기면 `orders -> order_items -> products` N+1 문제가 발생할 수 있다.
- 집계 쿼리에 fetch join을 적용하는 것은 보통 적합하지 않다.

전략:

| 상황 | 전략 |
| --- | --- |
| 단건 주문 상세 조회 | 필요한 경우 fetch join 또는 EntityGraph 검토 |
| 주문 목록 조회 | DTO projection, batch size, fetch join pagination 주의 |
| 인기 상품 집계 | DTO projection + 인덱스 + 사전 집계/캐시 검토 |
| 상품 상세에서 주문 목록 필요 | 별도 쿼리로 분리, 무조건 양방향 순회 금지 |

수업 포인트:

- fetch join은 연관 Entity를 함께 가져오는 전략이다.
- group by 집계 쿼리의 핵심 병목은 row 필터링, 조인, 집계, 정렬이다.
- `findPopularProducts`에는 fetch join보다 인덱스, 집계 테이블, 캐시가 더 직접적인 최적화 후보이다.
